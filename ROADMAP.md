# ROADMAP доработок логики загрузки и схлопывания

## Цель
Привести поведение пайплайна к правилам:
- загружать все строки из всех таблиц и стендов без исключений;
- сохранять дубли (включая дубли по ключу внутри одного файла);
- схлопывать только полностью идентичные строки по ключу;
- корректно учитывать доп. колонки при сравнении;
- не удалять строки, если по одному ключу есть различия в данных.

## План работ

- [v] 1. Зафиксировать текущие расхождения между требованием и кодом
  - Проверить этапы `_parse_all_rows`, `_merge_entity_default`, `_merge_entity_with_optional_fields`.
  - Подтвердить, что сейчас в таблицу `raw_rows` пишутся только строки из файлов со статусом `NEW`, а не все прочитанные строки.
  - Подтвердить, что сравнение/схлопывание в режиме `optional_fields.enabled=true` выполняется через кластеризацию по совместимости, а не по строгой идентичности набора значений.

- [v] 2. Исправить логику загрузки в SQLite (`raw_rows`)
  - Убрать условие записи только для `NEW`-файлов.
  - Писать все прочитанные строки каждого запуска в `raw_rows` (с привязкой к `run_id`), включая дубли.
  - Оставить `ingested_files` как журнал хэшей и статусов, но не как фильтр данных для `raw_rows`.

- [v] 3. Ввести единые правила "строки идентичны/разные" для схлопывания
  - Сначала группировать по `business_key`.
  - Внутри ключа схлопывать только те строки, которые идентичны по значимым полям.
  - Если строки по одному ключу различаются хотя бы в одном сравниваемом поле, хранить их отдельными merged-строками.

- [v] 4. Привести обработку доп. колонок к требуемому поведению
  - Для сравнения пары строк учитывать только пересечение наборов колонок.
  - Если колонка присутствует только в одной строке и отсутствует в другой, это не считать отличием.
  - При схлопывании объединять набор колонок (union), чтобы доп. колонка попадала в итоговую merged-строку.

- [v] 5. Обновить формирование `source_stands`
  - Формировать метку источников по фактическим строкам, вошедшим в конкретный merged-кластер.
  - Сохранять стабильный порядок `PROM-IFT-PSI` (или конфигурируемый порядок из `stands`/`reference_row_stand`).
  - Для внутристендовых дублей не терять информацию о том, что это один и тот же стенд (в `source_count` считать уникальные стенды, а дубли строк сохранять на уровне raw/кластеров, если они не идентичны).

- [v] 6. Уточнить выбор строки-эталона для вывода значений
  - Выбирать базовую строку из `PROM` при наличии, иначе по текущему рангу стендов.
  - При этом объединять недостающие поля из других строк кластера, чтобы не терять доп. колонки.

- [v] 7. Добавить тесты на критичные сценарии в `src/Tests`
  - Идентичные строки на `PROM` и `IFT` -> 1 merged-строка, `source_stands=PROM-IFT`.
  - Один ключ, но разные значения внутри одного файла -> несколько merged-строк.
  - Один ключ, разные значения между стендами -> несколько merged-строк.
  - Доп. колонка только на одном стенде -> строки считаются идентичными, колонка сохраняется в merged.
  - Полные дубли (одинаковые ключ+значения) в исходных данных -> в merged схлопываются, в `raw_rows` остаются все.

- [w] 8. Расширить логирование и диагностику
  - Добавить DEBUG-сообщения о причинах разделения строк по кластерам.
  - Логировать метрики: сколько строк прочитано, сколько merged-кластеров, сколько схлопнуто как идентичные, сколько оставлено как разные.

- [w] 9. Валидация результата
  - Прогнать пайплайн на текущем наборе CSV.
  - Проверить вручную несколько ключей с дублями и расхождениями.
  - Проверить корректность листов сущностей, `DIFF_REPORT`, `raw_rows`, `merged_rows`.
