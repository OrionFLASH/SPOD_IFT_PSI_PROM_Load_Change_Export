# Roadmap: консолидация настроек SPOD

Документ отражает **плановые этапы**, **фактическую реализацию** и **бэклог**.

## Формат статусов для каждого пункта

| Маркер | Значение |
|--------|----------|
| `[v]` | Сделано |
| `[w]` | В работе / требует доработки |
| `[ ]` | Не сделано |
| `[x]` | Отменено |

В списках используется запись **`- [v]`**, **`- [w]`**, **`- [ ]`**, **`- [x]`** (пробел в незакрытом чекбоксе обязателен). Правило для Cursor: `.cursor/rules/roadmap-spod-status.mdc`.

## План: проверки консистентности (анализ SPOD_PARCE_LOAD)

**Источник:** репозиторий [OrionFLASH/SPOD_PARCE_LOAD](https://github.com/OrionFLASH/SPOD_PARCE_LOAD) — в `config.json` секция **`consistency_checks`**: сводный лист (у них `CONSISTENCY`), подсекция **`csv_columns_count`**, массив **`rules`** с полем **`type`**.

Ниже — что **можно перенести в текущий проект** в первую очередь: логика опирается на **уже читаемые сырые строки CSV** (и заголовки), **без добавления новых колонок в источники**; расширяется только **`config.json`** и код пайплайна / отчёта.

### Приоритет 1 — сырые данные + конфиг, без SPOD-JSON в ячейках

| ID плана | Проверка (как в PARCE_LOAD) | Суть для текущего экспортера | Статус |
|----------|-----------------------------|------------------------------|--------|
| C-01 | **`csv_columns_count`** | По каждой сущности и стенду: число полей в заголовке и строках согласовано с ожидаемым из конфига или эталоном (`0` = max числа колонок заголовка по стендам). Лист **`CONSISTENCY`**, колонка статуса на листе сущности из `csv_columns_count.output`. | `[v]` |
| C-02 | **`unique`** (формализация) | Правила в `consistency_checks.rules`: `key_columns` или `use_business_key` для merged; `scope` **per_stand** / **merged** / **both**; WARNING при дубле в файле по-прежнему в парсере. | `[v]` |
| C-03 | **`field_length`** | Лимиты длины по колонкам из конфига; `scope` per_stand / merged / both. | `[v]` |
| C-04 | **`field_format`** (подтипы **date** / **decimal** / **fixed_length_digits**) | Проверка формата сырой ячейки по конфигу. | `[v]` |
| C-05 | **`referential`** | Ссылочная целостность по конфигу; per_stand (цель на том же стенде) и merged (цель — объединение всех стендов). | `[v]` |

### Приоритет 2 — больше конфигурации или вторая фаза отчёта

| ID плана | Проверка | Суть | Статус |
|----------|----------|------|--------|
| C-06 | **`referential_composite`** | Реализовано через тот же движок, что и `referential`, с `source_columns` / `target_key_columns`. | `[v]` |
| C-07 | **`cross_sheet_date_lte_today`** | Колонка + `date_format`; сравнение с локальной датой «сегодня» (без отдельной TZ в конфиге — **улучшение:** вынести часовой пояс в конфиг при необходимости). | `[v]` |

### Приоритет 3 — тяжёлый перенос (JSON в ячейках как в PARCE_LOAD)

Имеет смысл **после** появления общего слоя отчёта консистентности (лист или файл), если нужны те же семантики, что в PARCE_LOAD:

| ID плана | Проверка | Комментарий | Статус |
|----------|----------|-------------|--------|
| C-08 | **`json_spod_format`** | Парсинг JSON в ячейке, нормализация тройных кавычек как в SPOD-CSV; `json_required`. | `[v]` |
| C-09 | **`json_field_equals_column`** / **`json_field_in_column`** | Сравнение поля по `json_path` с колонкой или со списком в колонке. | `[v]` |
| C-10 | **`json_priority_unique_per_contest_link`** | Упрощённая проверка уникальности поля внутри JSON-массива в строке (**улучшение:** полная семантика PARCE под домен наград при появлении требований). | `[v]` |

### Не переносить в тот же вид / вне скоупа «сырые CSV»

- В PARCE_LOAD есть вспомогательные типы вокруг **обработанного Excel** (например колонки результата на листе); здесь вход — **три набора CSV**, выход — merge + Excel. Правила **C-01…C-06** проектируются как этап **после парсинга ParsedRow**, до или вместе с merge, плюс опциональный блок в **SUMMARY** / отдельный лист **`CONSISTENCY`** по аналогии с PARCE_LOAD.
- Тип **`pad_left_zeros`** в их конфиге относится к другим блокам (не к ядру `consistency_checks.rules` в выборке выше) — при переносе уточнять по месту.

**Статус переноса:** реализовано в `src/spod_exporter/consistency_checks.py`, интеграция в `pipeline.py`, примеры правил в `config.json`. Тяжёлые или доменно-спорные правила в конфиге по умолчанию с **`enabled: false`** — включайте по контуру данных.

| Задача | Статус |
|--------|--------|
| Интеграционные тесты на больших наборах CSV (в репозитории или CI) | `[ ]` |

| CI-пайплайн (unit-тесты на каждый push) | `[ ]` |
| Миграции / версионирование схемы SQLite | `[ ]` |
| Экспорт технического отчёта в JSON/CSV отдельно от Excel | `[ ]` |
| Регламент запуска, архивирование логов/запусков, шаблоны `config` под контуры | `[w]` — описано в плане работ; автоматизация и шаблоны контуров не внедрены |

Пояснение к последней строке: данные для анализа полей уже выводятся в Excel; отдельный интерактивный слой не делался. При осознанном отказе от UX задачу можно перевести в `[x]` с комментарием.
