# Roadmap: консолидация настроек SPOD

Документ отражает **плановые этапы**, **фактическую реализацию** и **бэклог**.

## Формат статусов для каждого пункта

| Маркер | Значение |
|--------|----------|
| `[v]` | Сделано |
| `[w]` | В работе / требует доработки |
| `[ ]` | Не сделано |
| `[x]` | Отменено |

В списках используется запись **`- [v]`**, **`- [w]`**, **`- [ ]`**, **`- [x]`** (пробел в незакрытом чекбоксе обязателен). Правило для Cursor: `.cursor/rules/roadmap-spod-status.mdc`.

---

## Общий статус (сводка)

| Область | Статус |
|---------|--------|
| ETL (сканирование, парсинг, merge, Excel, SQLite) | `[v]` |
| Конфигурация `config.json` (пути, сущности, форматирование, runtime) | `[v]` |
| Pre-check обязательных файлов | `[v]` |
| Параллельная обработка и `dry-run` | `[v]` |
| Лист `DIFF_REPORT` | `[v]` |
| Unit-тесты в репозитории | `[w]` — есть 3 теста в `src/Tests/test_pipeline.py`, покрытие неполное |
| Автоматический CI (GitHub Actions и т.п.) | `[ ]` |
| Формальные миграции схемы БД | `[ ]` |
| Валидация схемы CSV по обязательным колонкам | `[ ]` |

## Этапы 0–9 (детализация)

### Этап 0. Формализация ТЗ

**Статус этапа:** `[v]`

- [v] Зафиксированы сущности (10 типов), ключи, правила эквивалентности и optional-поля.
- [v] Текст ТЗ вынесен в `Docs/ТЗ_SPOD_Консолидация.md`.

---

### Этап 1. Каркас проекта

**Статус этапа:** `[v]`

- [v] Модули: `config`, `logging_setup`, `pipeline`, `models`, `main`.
- [v] Структура каталогов `src`, `src/Tests`, `IN`, `OUT`, `log`, `Docs`.

---

### Этап 2. Инжест

**Статус этапа:** `[v]`

- [v] Чтение CSV из трёх стендов (`IFT`, `PROM`, `PSI`), расчёт `SHA-256`, регистрация в SQLite.
- [v] Имена файлов задаются в `config.json` отдельно по стенду и сущности.
- [v] Pre-check: отсутствие обязательных файлов/каталогов блокирует запуск с перечислением пропусков.
- [v] Пакетная запись в БД (`executemany`) для производительности.

---

### Этап 3. Нормализация и сравнение

**Статус этапа:** `[v]`

- [v] Нормализация строк, `row_hash`, `business_key`, fallback-ключ `HASH:*`.
- [v] Флаги `same_key_diff_values_flag`, `same_key_diff_values_note`.
- [v] Optional-поля для выбранных сущностей (например `REPORT`, `EMPLOYEE`): эталон `reference_stand`, сравнение только если поле есть в обеих строках.
- [v] Служебные поля diff в выводе и конфиге: `diff_columns`, `diff_positions`, `diff_snippets` (часть в `service_fields` / настройках листа).

---

### Этап 4. Формирование merged-набора

**Статус этапа:** `[v]`

- [v] `source_stands`, `source_count`, `is_equal_all`, `diff_group_key`.
- [v] Разделение строк при коллизии `row_hash` при разном «сыром» наборе полей; эталон значений для отображения — `merge.reference_row_stand` (по умолчанию `PROM`).
- [v] Расширенная статистика разночтений в консоли и логах.

---

### Этап 5. Экспорт Excel

**Статус этапа:** `[v]`

- [v] Лист `SUMMARY` и по одному листу на сущность (10 типов).
- [v] Закрепление областей, автофильтр, автоподбор ширины с лимитом, перенос текста, выравнивание.
- [v] Рамки данных (пунктир/точки), двойной разделитель под заголовком, толстая вертикальная граница CSV / служебного блока.
- [v] Подсветка строк (в т.ч. отсутствие в `PROM`, конфликт значений) с приоритетом из конфига.
- [v] Подсветка заголовков ключевых полей и доп.полей (зелёный для доп.).
- [v] Выход в `OUT/XLS`, имя файла: `output_name_prefix` + `output_timestamp_format`.
- [v] Лист **`DIFF_REPORT`** (включается через `excel.diff_report_sheet.enabled` в `config.json`) — агрегированная сводка конфликтов.

---

### Этап 6. Логирование и отчёты

**Статус этапа:** `[v]`

- [v] INFO/DEBUG логи с шаблоном имён и форматом DEBUG-строки по ТЗ.
- [v] Потоковый консольный отчёт: старт, `run_id`, файлы, прогресс, БД, совпадения/разночтения.
- [v] Профилирование времени этапов пайплайна в логах.

---

### Этап 7. Производительность и режимы запуска

**Статус этапа:** `[v]`

- [v] Параллельная обработка (`ThreadPoolExecutor`), число потоков: авто по CPU или `runtime.parallel_workers` / CLI `--parallel-workers`.
- [v] Режим `dry-run`: без записи raw/merged в SQLite и без сохранения Excel (через `config.json` и флаг `--dry-run`).

---

### Этап 8. Тестирование

**Статус этапа:** `[w]` — базовые и ручные регрессии есть, CI и полное покрытие — нет

- [v] Unit-тесты: формирование `business_key`, fallback по hash, сценарий merge при одинаковом `row_hash` и разном сыром содержимом (`src/Tests/test_pipeline.py`, 3 теста).
- [v] Интеграционные и регрессионные прогоны на данных (сценарии S0–S4), результаты в `Docs/TestReports/` (в т.ч. `test_results_regression_latest.md`).
- [ ] Автозапуск интеграционных сценариев в CI при каждом коммите.
- [ ] Unit-тесты на правила подсветки Excel и все ветки форматирования из конфига.
- [ ] Интеграционные тесты на «больших» наборах CSV внутри репозитория (без внешних данных).

---

### Этап 9. Документация

**Статус этапа:** `[v]` *(поддерживается в актуальном виде)*

- [v] `README.md` — задача, структура, запуск, конфиг, логи, тесты, история версий.
- [v] `Docs/ТЗ_SPOD_Консолидация.md`, `Docs/Системные_требования_SPOD.md`, `Docs/План_работ_SPOD.md`, отчёты в `Docs/TestReports/`.
- [v] Настоящий roadmap со статусами `[v]` / `[w]` / `[ ]` / `[x]` по правилу `.cursor/rules/roadmap-spod-status.mdc`.

## Бэклог (после этапа 9)

| Задача | Статус |
|--------|--------|
| Интеграционные тесты на больших наборах CSV (в репозитории или CI) | `[ ]` |
| Валидация схемы входных CSV по обязательным колонкам для каждой сущности | `[ ]` |
| CI-пайплайн (unit-тесты на каждый push) | `[ ]` |
| Миграции / версионирование схемы SQLite | `[ ]` |
| Экспорт технического отчёта в JSON/CSV отдельно от Excel | `[ ]` |
| Регламент запуска, архивирование логов/запусков, шаблоны `config` под контуры | `[w]` — описано в плане работ; автоматизация и шаблоны контуров не внедрены |
| Детализация «какие поля отличаются» для одного ключа в отдельном UX (сверх колонок diff и сниппетов в Excel) | `[w]` — в отчёте есть `diff_columns` и сниппеты; отдельного drill-down UI пока нет |

Пояснение к последней строке: данные для анализа полей уже выводятся в Excel; отдельный интерактивный слой не делался. При осознанном отказе от UX задачу можно перевести в `[x]` с комментарием.

---

## Пример использования `[x]` (отменено)

Строка переводится в **`[x]`**, если задача снята с плана осознанно (больше не будет делаться). Сейчас в бэклоге таких пунктов нет — маркер зафиксирован в легенде для единообразия.
