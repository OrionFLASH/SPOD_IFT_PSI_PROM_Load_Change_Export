# Техническое задание (ТЗ): Консолидация настроек SPOD

## 1. Назначение

Разработать программу, которая:
- загружает CSV-настройки из стендов `IFT`, `PROM`, `PSI` из каталога `IN/SPOD`;
- объединяет однотипные данные между стендами;
- формирует Excel-отчет;
- форматирует Excel по правилам из `config.json` (закрепление, фильтр, ширина, выравнивание, подсветка);
- сохраняет результаты и историю загрузок в SQLite;
- исключает повторную загрузку одинаковых файлов по `SHA-256`.

## 2. Входные данные

Источник: `IN/SPOD/{IFT|PROM|PSI}`.

Набор сущностей (10 типов):
- `CONTEST`
- `EMPLOYEE`
- `GROUP`
- `INDICATOR`
- `ORG_UNIT_V20`
- `REPORT`
- `REWARD`
- `REWARD-LINK`
- `SCHEDULE`
- `USER_ROLE`

## 3. Правила обработки

1. CSV читаются в исходном виде (разделитель `;`, без изменения структуры столбцов).
   - Имя файла для каждой сущности задается отдельно по стендам в `config.json` (`file_names.IFT/PROM/PSI`).
   - Перед стартом выполняется проверка наличия всех обязательных файлов; при отсутствии хотя бы одного файла процесс не запускается, выводится полный список отсутствующих файлов и каталогов.
2. Для каждой строки формируются:
   - `business_key` по сущности;
   - `row_hash` по каноническому JSON строки.
   - для сущностей с `optional_fields.enabled=true` сравнение выполняется по правилам доп.полей:
     - эталон по составу полей — `reference_stand` (`PROM`);
     - отсутствие доп.поля не считается отличием;
     - если доп.поле есть в обоих сравниваемых файлах, его значение учитывается.
3. Объединение выполняется по `(business_key, row_hash)`:
   - если строки эквивалентны во всех 3 стендах -> одна запись с `source_stands=PROM-IFT-PSI`;
   - если отличаются -> отдельные записи с маркировкой `PROM`/`IFT`/`PSI`.
4. В SQLite ведется история запусков и загрузок файлов.
5. Повторно не загружаются файлы, чьи hash уже есть в БД.
6. В консоль выводится расширенная статистика:
   - количество найденных файлов по каждой сущности;
   - количество обработанных файлов и строк;
   - сколько файлов добавлено в БД и сколько пропущено по совпадению hash;
   - количество merged-строк, совпадений и разночтений;
   - количество ключей, у которых есть разные значения в разных стендах.

## 4. Выходные данные

1. Excel в `OUT/XLS`:
   - 1 лист на сущность;
   - лист `SUMMARY`;
   - автофильтр на строке заголовка;
   - закрепление области (`freeze_panes`) из конфига;
   - автоподбор ширины с ограничением;
   - перенос текста в ячейках;
   - выравнивание заголовков и данных по настройкам;
   - цветовая подсветка строк по правилам.
   - рамки для строк данных:
     - горизонтальные границы — тонкий пунктир;
     - вертикальные границы — тонкие точки;
   - заголовок отделяется тонкой двойной линией;
   - служебные поля отделяются от блока исходных данных толстой горизонтальной линией;
   - цветовые правила применяются по приоритету из `config.json`.
   - заголовки доп.полей (которые есть за пределами эталона `PROM`) выделяются зеленым.
2. SQLite в `OUT/DB`:
   - таблицы `runs`, `ingested_files`, `raw_rows`, `merged_rows`.
3. Логи в `log`:
   - `INFO_*`
   - `DEBUG_*`

## 5. Обязательные служебные поля в merged-данных

- `source_stands`
- `source_count`
- `is_equal_all`
- `diff_group_key`
- `same_key_diff_values_flag`
- `same_key_diff_values_note`

## 6. Управление через конфиг

Параметры путей, сущностей, ключей, логирования, имен выходных файлов и runtime-режимов задаются в `config.json`.

Дополнительно в `config.json` задаются:
- имена файлов по каждому стенду для каждой сущности;
- параметры листа Excel (`freeze_panes`, `auto_filter_header`);
- параметры форматирования (`max_column_width`, перенос, выравнивание, цвета подсветки, рамки, приоритеты заливки, выделение ключевых заголовков);
- параметры обработки доп.полей (`optional_fields.enabled`, `optional_fields.reference_stand`);
- параметры имени выходного файла:
  - `output_name_prefix` (постоянная часть);
  - `output_timestamp_format` (формат таймштампа);
  - расширение `.xlsx` добавляется автоматически.
